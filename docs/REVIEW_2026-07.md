# Repo Review — July 2026

A housekeeping pass and honest health check of the archthesis repo, written in
plain language. Nothing in this pass changes how the live site behaves — it is
all documentation, organization, and bookkeeping.

---

## What I cleaned up

**1. Moved old session notes out of the root folder → `docs/archive/`**

The root of the repo had accumulated seven documents that were written during
specific work sessions and then never updated: two write-ups of the same
placeholder-text bug fix, a deployment record for the origin-tracking feature,
a project status snapshot from January, and three translation working
documents. They were useful the day they were written, but by now their dates,
version numbers, and file references are out of date, and they made the root
folder look busier than the project actually is.

They were **moved, not deleted** — everything is still in `docs/archive/`,
with a small README there explaining what each file was. Nothing was lost.
This is purely a filing change; the app itself is untouched.

**2. Rewrote the README**

The old README described the project as mid-migration ("Week 2 in progress,
canvas editor upcoming") when in reality the site has been live in production
since January. It also pointed to a `SESSION_NOTES.md` file that no longer
exists, and listed library versions that no longer match what's installed.
The new README describes the project as it is today: what it is, where it's
live, how to set it up (including the required `.env` file), the real folder
structure, and where the documentation lives.

**3. Stopped tracking a build cache file (`.firebase/hosting.ZGlzdA.cache`)**

This file is generated automatically every time you deploy — it's a scratch
file, not source code, and it would have shown up as a meaningless change in
every future deploy. It's now ignored by git (along with test-coverage output
and other generated folders). The file still exists on disk; git just stops
watching it.

**4. Deleted two genuinely unused files**

`src/App.css` and `src/assets/react.svg` are leftovers from the project
template that created the app — I verified nothing in the code references
either of them. Also removed four empty `.gitkeep` placeholder files whose
folders have long since filled up with real code. Zero risk: none of these
were used anywhere.

**5. Moved `FIREBASE_SETUP_GUIDE.md` into `docs/`**

It's a living, still-useful guide — it just didn't need to be in the root.
The README links to its new home.

## What's in good shape

Honest credit where it's due — several things here are genuinely solid:

- **No secrets in the repo.** Firebase credentials come from environment
  variables, `.env` is properly git-ignored, and `.env.example` shows the
  shape without the values. This is exactly how it should be done.
- **The build protects you from a known failure mode.** `scripts/check-env.mjs`
  makes the build fail loudly if Firebase credentials are missing, instead of
  shipping a broken white-screen site. That guard was added after a real
  incident and it's a good one.
- **Security rules are thoughtful.** The Firestore rules let anonymous users
  create memes but cap every text field's length, only allow the public to
  change the "likes" counter, and reserve deletion for the admin. Storage
  limits uploads to images under 20MB. Admin status uses Firebase custom
  claims rather than a hardcoded email in the rules. For a solo project, this
  is above average.
- **The CHANGELOG is a real asset.** It's detailed, newest-first, and matches
  the actual version in `package.json` (3.3.8). Keep feeding it.
- **There are actual tests** (component tests, store tests, an admin-auth
  integration test) and the tooling to run them. Many solo projects have none.
- **Performance basics are handled:** pages load lazily, vendor code is split
  into chunks, images get long cache headers, and there's a version checker
  that tells users when a new deploy is available.

## Suggestions for the future

In priority order. None of these are emergencies.

**1. Replace the placeholder favicon.** *(Low effort, visible win.)*
The browser-tab icon is still Vite's default logo (`public/vite.svg`,
referenced in `index.html`). The site has real users and a real brand — a
proper favicon (and matching PWA icons) is a five-minute design task that
makes the site look finished. No risk.

**2. Decide the fate of the "no-cache" meta tags in `index.html`.**
*(Small, but worth understanding.)* The HTML asks browsers to never cache the
page, twice: once in meta tags and once in Firebase hosting headers. The
hosting headers are the ones that actually matter; the meta tags are mostly
ignored by modern browsers and can confuse future readers. Removing them
would be a cleanup, but since caching bugs were painful here in the past, I
deliberately left them alone — behavior changes were out of scope for this
pass. If you ever revisit caching, do it in one place (the `firebase.json`
headers) and delete the meta tags.

**3. Set up an automated safety net (CI).** *(Medium effort, biggest
long-term protection.)* Right now, tests and the linter only run if someone
remembers to run them. A small GitHub Actions workflow could automatically
run `npm run lint`, `npm test`, and a production build on every pull request.
Benefit: you'd find out about breakage *before* it merges, which matters
especially because you review changes at a surface level and trust the model.
Risk: essentially none — it only reads the code, it can't break the live site.

**4. Finish the English translation, or clearly park it.** *(Scope decision,
not a code task.)* The i18n foundation is merged and the gallery/home pages
are translated, but the editor and admin areas are still Hebrew-only. Either
finishing the pass or noting "Hebrew-only beyond the gallery, by design" in
the README would prevent a half-translated impression. The master
translation table is preserved in `docs/archive/TRANSLATION_MASTER.md`.

**5. Consider trimming the CHANGELOG's oldest entries into the archive.**
*(Cosmetic, someday.)* At ~1,500 lines it's getting long. Keeping the last
few versions at the top and moving ancient history into `docs/archive/`
would make it friendlier. Purely optional.

## Things to watch out for

Honest risk flags — things I noticed but deliberately did **not** change,
because changing them would touch runtime behavior:

- **Anyone can publish, and anyone can inflate likes.** By design, anonymous
  users can create memes, and the security rules allow anyone to update the
  `likes` field of any meme — with no rate limit and no check that it only
  increments by one. A motivated prankster could set any meme's like count to
  any number. App Check (bot protection) raises the bar, but doesn't stop a
  determined human. For a thesis project this is probably an acceptable
  trade-off — just don't treat like-counts as trustworthy research data.
- **The like-count caveat extends to research use.** If gallery analytics
  feed the thesis, note somewhere in the methodology that engagement numbers
  are unauthenticated and tamperable.
- **`src/version.ts` is updated by hand.** The version banner users see comes
  from this file, while `package.json` has its own version number. Today both
  say 3.3.8, but they can drift apart silently. If they do, the
  "new version available" notification may behave oddly. A future improvement
  would be to derive one from the other.
- **Firestore duplicate rules note:** the `templates` and `analytics`
  collections are declared in the rules but marked "future use" — if they're
  never adopted, that's fine, just remember they exist before assuming the
  rules file describes reality.
- **No CI means regressions ride on memory.** Until suggestion #3 above is
  in place, the only thing standing between a bad change and production is
  remembering to run the tests and the build locally.
- **This pass did not audit application logic.** I read the code at the
  structural level (what exists, what references what) and the security rules
  in full, but I did not line-by-line review the editor or publishing logic.
  "Nothing jumped out" is not the same as "verified correct."
